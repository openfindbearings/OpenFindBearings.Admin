# 映射维护模块 v1.0.0

## 概述

映射维护模块管理从原始数据（爬虫抓取、手动录入）到标准化数据的映射关系。核心是两个映射表：品牌名称到 Code、类型名称到 Code。T 阶段使用这些映射做标准化转换。

## 映射关系说明

### 品牌名称 → Code

CBIA 爬虫抓到的品牌名可能是中文（"天马"）、英文（"TMB"）、混合或错别字。映射表将各种写法归一化为标准 Code：

```
┌───────────────────────────────────────────────────┐
│ 原始名称          │ 标准 Code │ 来源     │ 可信度   │
├───────────────────────────────────────────────────┤
│ TMB              │ TMB      │ CBIA     │ 100      │
│ 天马              │ TMB      │ 自动匹配  │ 85       │
│ 天马轴承          │ TMB      │ 手动确认  │ 100      │
│ TIANMA BEARING   │ TMB      │ 自动匹配  │ 80       │
│ NSK              │ NSK      │ CBIA     │ 100      │
│ 日本精工          │ NSK      │ 自动匹配  │ 80       │
│ 斯凯孚(SKF)      │ SKF      │ 手动确认  │ 100      │
│ SKF              │ SKF      │ CBIA     │ 100      │
└───────────────────────────────────────────────────┘
```

### 类型名称 → Code

类型名称同理：

```
┌───────────────────────────────────────────────────┐
│ 原始名称            │ 标准 Code │ 来源     │ 可信度   │
├───────────────────────────────────────────────────┤
│ 圆柱滚子轴承        │ CRB      │ CBIA     │ 100      │
│ 圆柱滚子            │ CRB      │ 自动匹配  │ 85       │
│ Cylindrical Roller │ CRB      │ 自动匹配  │ 80       │
│ 深沟球轴承          │ AB       │ CBIA     │ 100      │
│ Deep Groove Ball   │ AB       │ 自动匹配  │ 80       │
└───────────────────────────────────────────────────┘
```

## API 调用

映射数据存储位置有两种方案：

| 方案 | 存储位置 | 优缺点 |
|------|---------|--------|
| A: Sync 数据库 | 存在 staging 层的映射表 | Sync 直接使用，但 Admin 需直连 db_sync |
| B: Admin 独立管理 | Admin 自己通过 API 维护 | 解耦，但 Sync 需调用 Admin 接口获取 |

推荐**方案 B**——Admin 通过 OpenFindBearings.Api 的同步端点（`/api/sync/brands/batch` 和 `/api/sync/bearing-types/batch`）直接维护品牌和类型数据，映射逻辑在 Admin 侧管理。

具体来说：

1. Admin 展示所有已存在的标准 Code（通过 BusinessApiClient 查询 API）
2. Admin 展示所有原始名称（通过 CrawlerApiClient 查询爬虫数据）
3. 管理员手动关联原始名称到标准 Code
4. 映射结果存入 Admin 配置或独立映射表（可以通过 API 的扩展端点）

## 导入/导出

- 导入 CSV：批量导入映射关系（字段：原始名称, 标准Code, 来源）
- 导出 CSV：导出全部映射关系
- 导入时自动去重和校验 Code 是否存在

## 审核工作流

```
1. T 阶段自动匹配 → 可信度 >= 85 → 自动确认
2. 可信度 60-84 → 标记为"待审核"
3. 可信度 < 60 或未匹配成功 → 标记为"未映射"
4. 管理员在 Admin 页面审核待处理和未映射条目
5. 手动确认后可信度提升到 100
6. 已自动确认但管理员修改的 → 来源标记为"手动修正"
```

## 可信度阈值配置

T 阶段的自动匹配行为由一组阈值参数控制，管理员可通过 Admin 调整这些参数（配置持久化到 Sync 的 config 表或 Admin 的 AdminConfigs 表）。

### 参数说明

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `AutoSyncThreshold` | 85 | 可信度 >= 此值自动确认，无需审核 |
| `PendingReviewThreshold` | 60 | 可信度 >= 此值但 < AutoSyncThreshold → 待审核 |
| `RejectThreshold` | 60 | 可信度 < 此值 → 标记为未映射，必须人工处理 |
| `BrandMatchScoreDefault` | 50 | 新品牌匹配器的默认基础可信度 |
| `ExactMatchBonus` | 50 | 品牌名精确匹配时额外加分（50+50=100） |
| `FuzzyMatchBonus` | 30 | 模糊匹配时额外加分（50+30=80） |

### 配置页面

```
┌─── 可信度阈值配置 ───────────────────────────┐
│                                               │
│ 自动确认阈值 (AutoSyncThreshold): [85   ]     │
│ 待审核阈值 (PendingReviewThreshold): [60  ]    │
│ 拒绝阈值 (RejectThreshold): [60         ]     │
│ 品牌匹配基础分 (BrandMatchScoreDefault): [50] │
│ 精确匹配加分 (ExactMatchBonus): [50      ]    │
│ 模糊匹配加分 (FuzzyMatchBonus): [30      ]    │
│                                               │
│ [保存] [恢复默认]                               │
│                                               │
│ 提示: 修改阈值影响 T 阶段自动审核行为，          │
│ 修改后下一次 Transform 执行生效。               │
└───────────────────────────────────────────────┘
```

调整阈值的影响：
- 提高 AutoSyncThreshold（如 90）→ 更多记录需人工审核，降低误匹配风险
- 降低 AutoSyncThreshold（如 70）→ 更多记录自动通过，减少人工工作量
- 调整匹配加分 → 控制品牌/类型名称模糊匹配的宽松程度
